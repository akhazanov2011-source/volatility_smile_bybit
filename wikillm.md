# WikiLLM: Volatility Smile Bybit

## 1. Что это за проект

Небольшой Python-проект для визуализации `volatility smile` по опционам Bybit через публичный API V5.

Проект теперь устроен максимально просто:

- `app.py` — единственный Python-файл проекта. В нём находятся Flask-сервис, загрузка опционных тикеров Bybit, парсинг символов, фильтрация данных, подготовка smile-структуры и построение Plotly-графика для `IV`, `theta`, `theta / mark price (%)`, `mark price` и греков.
- `templates/index.html` — единственный HTML-шаблон интерфейса.
- `README.md` — актуальные сценарии запуска и описание проекта.
- `Dockerfile` — контейнерный запуск через Gunicorn.

CLI-режим, Dash-режим и отдельные Python-модули удалены. Проект работает только как веб-сервис.

## 2. Текущее фактическое состояние

- В репозитории остался один Python-файл: `app.py`.
- Flask-входная точка `app.py` импортируется и запускается корректно (Flask 3.x, без удалённого `before_first_request`).
- Docker entrypoint `gunicorn app:app` согласован с кодом; шаблон `templates/index.html` попадает в образ (`.dockerignore` больше не исключает `*.html`).
- Запросы к Bybit идут с таймаутом и более явной диагностикой ошибок сети/API.
- Внешняя сеть всё ещё обязательна для реального построения графика.
- Внешних зависимостей минимум: `Flask`, `gunicorn`, `plotly`, `requests`. APScheduler больше не используется — фоновый цикл реализован на `threading`.

## 3. Основной пользовательский сценарий

Пользователь открывает веб-страницу и выбирает монету, после чего проект:

1. Загружает опционные тикеры Bybit для выбранной монеты.
2. Извлекает `indexPrice` как spot.
3. Автоматически выбирает страйковое окно, покрывающее 70% ближайших к `spot` страйков.
4. Группирует опционы по экспирации и по типу `Call`/`Put`.
5. Оставляет OTM-крылья и по одной ближайшей ATM/ITM точке для визуального “моста”.
6. Строит Plotly figure, где для каждой экспирации улыбка рисуется одной сплошной линией, а пользователь может переключать метрику между `IV`, `theta`, `theta / mark price (%)`, `mark price` и греками.
7. Встраивает график в HTML через `fig.to_html(...)`.

## 4. Ключевые сущности в `app.py`

- `SUPPORTED_COINS` — список поддерживаемых активов: `BTC`, `ETH`, `SOL`, `XRP`, `DOGE`, `XAUTUSDT`.
- `SUPPORTED_METRICS` — доступные режимы графика: `iv`, `theta`, `theta_pct`, `mark_price` и греки.
- `COIN_ALIASES` — маппинг UI-имени актива на `baseCoin` Bybit и допустимые префиксы опционных символов.
- `parse_symbol(symbol)` — парсит bybit-символ вида `BTC-13APR26-72500-C-USDT`.
- `get_tickers(base_coin)` — делает запрос к `https://api.bybit.com/v5/market/tickers` с `category=option`, таймаутом, ретраями и валидацией ответа; результат кешируется на 60 с.
- `collect_strikes(...)` — собирает доступные страйки для выбранного актива.
- `get_strike_window(...)` — выбирает 70% ближайших к `spot` страйков.
- `fetch_and_prepare_data(...)` — центральная функция подготовки данных.
- `build_figure(...)` — строит интерактивный Plotly-график.
- `normalize_metric(...)` — валидирует выбранную пользователем метрику.
- `calculate_theta_pct(...)` — считает `theta / mark price * 100`.
- `_build_chart_entry(...)` — полный pipeline одной пары `(coin, metric)`: сеть → подготовка → рендер → dict записи кеша. Никогда не бросает исключение наружу.
- `refresh_combo(...)` — собирает запись для пары и кладёт в `_cache`.
- `_background_worker_loop(...)` / `_ensure_worker_started()` — фоновый поток-демон, обновляющий все пары каждые 3 минуты.
- `GET /healthz` — простая проверка живости.
- `GET /` — основная страница: читает кеш, на промахе синхронно обновляет пару.

## 5. Архитектурные особенности

### Один Python-файл

Вся бизнес-логика и веб-слой находятся в `app.py`. Это упрощает сопровождение и соответствует текущей цели проекта, но смешивает:

- доступ к внешнему API;
- преобразование данных;
- построение визуализации;
- Flask endpoint'ы.

### Нет локального состояния

Проект не использует:

- базу данных;
- файловое хранилище данных;
- аутентификацию;
- конфигурационный слой через `.env`.

### Есть фоновый in-memory кеш

Проект использует in-memory кеширование для скорости отклика:

- `_tickers_cache` — TTL-кеш сырых ответов Bybit (60 с), потокобезопасный.
- `_cache` — потокобезопасный кеш отрендеренного `chart_html` + статистики по ключу `(coin, metric)`.
- Фоновый поток-демон `_background_worker_loop()` каждые 3 минуты обновляет все пары `(coin, metric)`.
- Маршрут `/` читает кеш; при холодном старте/промахе синхронно обновляет запрошенную пару (`refresh_combo(...)`).

Всё построение происходит из ответов Bybit, постоянного хранилища нет.

## 6. Данные и их преобразование

Источник данных: публичный REST API Bybit V5:

- endpoint: `/v5/market/tickers`
- query params: `category=option`, `baseCoin=<coin>`

Из тикеров используются:

- `symbol`
- `markIv`
- `indexPrice`
- `markPrice`
- `delta`
- `gamma`
- `theta`
- `vega`

Фильтрация идёт так:

1. Пропускаются записи без `markIv`.
2. Символ опциона должен успешно парситься.
3. `base_coin` из символа должен совпадать с выбранной монетой.
4. Страйк должен попадать в автоматически рассчитанный интервал `[min_strike, max_strike]`.

Для каждой экспирации:

- `Call`: берутся OTM-точки со `strike > spot`
- `Put`: берутся OTM-точки со `strike < spot`
- затем добавляется ближайшая к `spot` ATM/ITM точка для визуального соединения крыла

## 7. Реальные точки входа

### Рабочее сейчас

- `python3.14 -m py_compile app.py`
- `python3.14 -c "import app"`

### Рабочее при наличии сети к Bybit

- `python3.14 app.py`
- `gunicorn app:app`
- `docker build -t volatility-smile-bybit .`
- `docker run --rm -p 8000:8000 volatility-smile-bybit`

## 9. Что важно учитывать LLM при изменениях

### 1. Ориентироваться на реальные файлы

Сейчас структура простая: почти вся ценная логика находится в `app.py`, а интерфейс — в `templates/index.html`.

### 2. Не возвращать удалённые режимы без явной причины

CLI-режим и Dash-режим удалены по решению пользователя. Если новая задача не просит вернуть их явно, проект нужно сохранять как чистый веб-сервис.

### 3. Визуализация зависит от доменной логики

`build_figure(...)` ожидает уже подготовленную структуру `by_expiry`. Если менять структуру данных, почти наверняка придётся менять hover, legend и фильтрацию.

### 4. Нет тестов, поэтому важна дешёвая верификация

Минимальная проверка после правок:

- `python3.14 -m py_compile app.py`
- smoke-import `app`
- если менялся веб-слой: Flask `test_client`
- если сеть доступна: ручной прогон Flask или Docker

## 10. Типовые задачи для будущих LLM

### Поддерживать веб-сервис

Самый вероятный сценарий:

- менять страницу `/`;
- улучшать сообщения об ошибках;
- развивать логику автоподбора страйков и представление графика;
- улучшать Docker-запуск.

### Добавить тесты

Наиболее полезные первые тесты:

- `parse_symbol(...)`
- `get_strike_window(...)`
- поведение фильтра по OTM/ATM в `fetch_and_prepare_data(...)`
- smoke-test Flask route через test client с моками сетевых функций

### Улучшить UX/надёжность

- обработка пустых/битых ответов API
- явные сообщения об ошибках сети
- кеширование spot/tickers
- развитие query string-параметров

## 11. Известные проблемы

- тестов в репозитории всё ещё нет
- проект не оформлен как Python package
- вся Python-логика сосредоточена в одном файле, что удобно сейчас, но может усложнить крупные доработки

## 12. Краткий вывод

Это маленький и понятный веб-проект для построения volatility smile по опционам Bybit. Главная ценность кода — в функции подготовки данных и построении Plotly figure. Основная особенность текущего состояния — всё собрано в одном `app.py`, без CLI и без дополнительных Python-модулей.
